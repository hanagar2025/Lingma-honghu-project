# 鸿鹄原架构 × TIOS新体系 · 深度评估与融合决策

> 评估日期：2026-07-19
> 方法：对 backend/src 全部 16 个服务、路由、数据层、定时任务、前端页面做逐文件审查。
> 结论先行：**两套框架在「决策哲学」上截然对立，但在「基础设施」上高度互补。推荐方案：同仓分舱——复用鸿鹄的地基，决策链路单轨切换到TIOS，预测型服务降级封存。不需要另起新程序。**

---

## 一、深查发现的事实

### 1.1 决策哲学：确实是两套截然不同的框架

| | 鸿鹄原架构 | TIOS新体系 |
|---|---|---|
| 核心问题 | 「哪只股票会涨？」 | 「今天触发了什么条款？」 |
| 输出 | buy/sell/increase + 预期收益 + 趋势预测 + 置信度 | 触发事件+条款编号，默认「今天什么也不做」 |
| 决策来源 | 技术/基本面/趋势多因子打分 | 预设规则→收盘确认→次日执行 |
| 代表模块 | decisionEngine、smartPositionRecommendation、portfolioStrategy、timeSeriesAnalysis、positionClassification（合计约6100行） | services/tios/*（约700行，本周新建） |
| 哲学定性 | **顾问式预测系统**——正是这次80万回撤中被证伪的工作方式 | **条款执行系统**——本周实盘验证过的工作方式 |

### 1.2 数据层：原架构的行情数据基本是假的（这是最关键的发现）

- `dataProvider.ts` 声明了 Tushare/东财/新浪三个数据源，**但代码中零调用**；
- 实时行情 `getStockQuote` **100%由 Math.random 生成**；K线、基本面在数据库为空时全部走 mock 生成器；
- 全后端 `Math.random` 出现约 **138处**，「模拟」注释约39处；
- 主决策链（打分推荐）约 **80-90% 依赖假数据**——意味着原系统的每一条「智能推荐」都建立在随机数上；
- 每小时「更新行情」的定时任务是**空实现（TODO）**。

**结论：原架构不是「预测得不好」，而是「从未真正接入过市场」。这反而降低了迁移成本——没有值得保留的决策历史。**

### 1.3 可复用清单（鸿鹄的真实资产）

- ✅ Express+TS 工程骨架、MySQL/Redis 连接层、JWT认证（登录/注册完整可用）
- ✅ 持仓管理 CRUD（positions表+API+前端页面）——闭环第①⑤步的现成载体
- ✅ 技术指标纯函数（MA/MACD/RSI等算法本身正确，只是输入是假的）
- ✅ 定时任务框架（cron，改挂TIOS任务即可）
- ✅ 前端框架（React+AntD+移动端壳、Layout、Login、Portfolio页）

### 1.4 冲突清单（必须下线的部分）

- ✗ decisionEngine / smartPositionRecommendation / portfolioStrategy / positionClassification / timeSeriesAnalysis（预测型五件套）
- ✗ 前端 Decisions / SmartRecommendation / PortfolioStrategy 三个页面
- ✗ reportGenerator 中「AI建议买卖」类叙事段落

## 二、三个方案的对比

| 方案 | 内容 | 优点 | 缺点 | 判定 |
|---|---|---|---|---|
| A. 完全融合 | TIOS嵌入原决策链，与打分推荐并行输出 | 改动最小 | **双决策源=体系自杀**：软件层面重演「每天两个声音」 | ❌ 否决 |
| B. 另起小程序 | 全新仓库，从零搭建 | 干净 | 重造auth/持仓/DB/前端≈2-3周纯基建工作；旧仓沦为弃船 | ❌ 不必要 |
| **C. 同仓分舱（推荐）** | 复用鸿鹄地基；TIOS为唯一决策链；预测五件套路由下线封存（代码保留不删） | 地基零重建；决策单轨；随时可对照旧代码 | 需一次「换脑手术」 | ✅ **采纳** |

**方案C的核心纪律：整个系统只允许存在一个决策生产者（TIOS盘前四问）。** 预测型服务的API路由注释下线（代码封存不删除），前端对应页面隐藏。这与投资体系「规则先于观点」同构——软件架构必须服从投资架构。

## 四、小规模试运行方案（下一步，P1）

**目标：用现有7只持仓，先跑通「数据→决策→执行→回填→复盘」一个完整闭环，每天一次。**

1. **数据接入（本周）**：申请Tushare token → `dataProvider` 接真实日K（只需7只持仓+2个基准指数，量极小）→ 收盘后自动入库 `stock_daily`
2. **规则卡入库（本周）**：7张规则卡（硬止损价/熔断阈值/禁买标记/补执行位）写入 `rule_cards` 表
3. **每日流水线（下周起）**：cron 17:00 → 拉收盘数据 → TIOS生成盘前四问 → 存 `decision_logs` → 前端新增「盘前四问」页展示
4. **执行回填**：你在券商App执行后，在持仓页录入成交 → 系统计算执行率/偏离
5. **周五**：领先指标人工录入页 + 周报自动生成（执行率、违规数、触发历史）

预测型五件套在本阶段路由下线。试运行一个月后（8月修订窗口+一个完整月度周期），评估是否扩展P2（估值自动化）。

*本文件为架构决策记录（ADR）。软件层面的一切设计服从投资体系的最高原则：单一决策源、规则先于观点、执行可追溯。*

## 五、决议更新（2026-07-19）：从「封存」升级为「删除」，精炼版落地

经决策，原「同仓分舱、封存不删」方案升级为**彻底删除**：预测型五件套（decisionEngine、smartPositionRecommendation、portfolioStrategy、sectorAnalysis、reportGenerator）及其全部支撑服务（技术/基本面/时序分析、websocket、systemOptimization、userExperience、mock版dataProvider）、对应路由与前端页面全部移除。系统只剩一个大脑：TIOS规则引擎。

实际落地与P1方案的差异：

1. **数据源**：未用Tushare（需token），改用**腾讯行情**（免费、无需注册）：实时报价 `qt.gtimg.cn`、前复权日K `web.ifzq.gtimg.cn`、股票搜索 `smartbox.gtimg.cn`。已实测可用。
2. **Redis 移除**：无缓存层，MySQL为唯一外部依赖，7张表覆盖闭环（users / positions / stock_daily / rule_cards / account_state / decision_reports / trade_executions）。
3. **闭环已跑通**（真实行情端到端验证）：同步320根真实日K → 刷新持仓市值 → 阶段判定（科创50基准）→ 盘前四问入库 → 卖出指令生成待执行记录 → 盘后回填 → 铁律执行率统计。
4. **定时任务**：工作日17:00（北京时间）自动运行全流程；页面可手动触发。
5. **前端只剩两页**：盘前四问（主页：决策单+执行回填+规则卡+账户状态）、持仓管理（含板块/主题字段供仓位闸门使用）。

至此，P1「小规模试运行」的软件部分全部就绪，剩余为使用层面：录入7只真实持仓与净值高点，开始每日执行与回填。
